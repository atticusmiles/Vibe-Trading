# 阶段 8：飞书集成

## 1. 概述

**目标**：对接飞书开放平台，实现提案推送卡片审批和飞书对话两种能力，提供移动端交互入口。

**与前后阶段的关系**：依赖阶段 4（提案机制）和阶段 2（用户 Session 体系）。本阶段为投研系统增加移动端触达能力。

**前置条件**：阶段 4 完成，提案审批 API 可用。

## 2. 飞书应用配置

每个用户自行配置：
- `settings.feishu.app_id`：飞书应用 App ID
- `settings.feishu.app_secret`：飞书应用 App Secret（加密存储）
- `settings.feishu.push_channel`：推送渠道（个人/群聊 ID）

系统级配置（所有用户共享同一个飞书应用，或各自独立应用）。

## 3. API 设计

```
POST   /api/feishu/webhook            飞书事件回调（消息事件 + 卡片操作回调）
POST   /api/feishu/bind               绑定飞书账号（传入飞书 user_id / open_id）
DELETE /api/feishu/bind               解绑飞书账号
GET    /api/feishu/config             获取飞书配置（脱敏）
PUT    /api/feishu/config             更新飞书配置
```

## 4. 业务逻辑

### 4.1 提案推送

新提案产生时（阶段 4 的提案创建逻辑），检查用户是否绑定飞书：
1. 已绑定 → 构造飞书交互卡片，发送到用户配置的推送渠道
2. 未绑定 → 跳过推送

卡片内容：
```
┌─────────────────────────────────┐
│  新提案：新增趋势"人民币走强"     │
│  类型：趋势 · 操作：新增          │
│  置信度：8/10                    │
│  摘要：基于XXX数据，判断...       │
├─────────────────────────────────┤
│  [  采纳  ]      [  拒绝  ]      │
└─────────────────────────────────┘
```

用户点击「采纳」或「拒绝」：
- 飞书发送卡片操作回调到 `/api/feishu/webhook`
- 系统调用阶段 4 的提案审批 API（adopt/reject）
- 更新卡片状态（显示"已采纳"或"已拒绝"）

### 4.2 飞书对话

用户在飞书中@机器人发消息：
1. 飞书推送消息事件到 `/api/feishu/webhook`
2. 系统根据飞书 user_id 查找绑定的用户
3. 创建或复用该用户的 Session
4. 将消息发送到 Session（复用现有 `POST /sessions/{id}/messages`）
5. Agent 执行完成后，将回复以飞书消息形式返回

对话上下文与 Web 端共享，用户可在飞书追问。

## 5. 验收标准

- [ ] 可通过 API 配置飞书应用信息（App ID/Secret）
- [ ] 可绑定/解绑飞书账号
- [ ] 新提案产生时自动推送飞书卡片
- [ ] 飞书卡片上点击采纳/拒绝，提案状态正确变更
- [ ] 飞书对话可发送消息并收到 Agent 回复
- [ ] 飞书对话与 Web 端共享 Session 上下文
