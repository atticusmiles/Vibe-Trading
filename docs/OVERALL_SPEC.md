# AI辅助投研系统 — 需求规格说明书

## 1. 产品概述

构建一个多 Agent 协同的 AI 辅助投研系统。系统通过多个专业化 Agent 协同工作，自动分析新闻流、行情数据、企业财报等信息，为用户提供具体股票的买卖操作建议。

## 2. 用户体系

### 2.1 用户角色与认证

- 系统的使用者为**投资者**，负责配置投资偏好、管理自选标的，获得结构化的投资建议报告
- 系统应支持多用户注册和登录，采用**用户名 + 密码**的认证方式
- 用户数据应**完全隔离**：每个用户只能看到自己的趋势、行业、股票、提案、投研历史等数据
- 用户认证信息应安全存储，不得明文保存

### 2.2 API Key 管理

- 每个用户应能独立配置自己的 API Key，运行投研任务时使用各自的额度
- 系统不提供任何默认 API Key，所有 Key 均由用户自行配置
- 需要支持的 Key 类型包括但不限于：

| 配置项 | 说明 |
|--------|------|
| LLM Provider Key | OpenRouter / OpenAI / DeepSeek / Gemini / DashScope 等 LLM 服务的 API Key |
| LLM 模型配置 | 使用的模型名称、Base URL、Provider 类型 |
| tushare token | A 股数据源认证 |
| searxng 地址 | 搜索引擎聚合服务的地址 |
| 其他第三方 Key | 后续扩展的数据源或服务密钥 |

- API Key 应加密存储，不得明文保存，前端展示时脱敏处理

## 3. 投研引擎

### 3.1 运行模式

系统支持以下两种运行模式：

| 模式 | 描述 | 交互方式 | 产出 |
|------|------|----------|------|
| 对话模式 | 用户以自然语言提问，Agent 团队协作分析后以对话形式返回结果 | 多轮对话，可追问、调整方向 | 即时对话回复，附带数据和分析依据 |
| 自动投研模式 | 系统按预设工作流自动执行完整投研流程，全程无需人工干预 | 启动后自动运行，可实时查看进度 | 结构化 Proposal（趋势/行业/股票的增删改提案 + 投资建议报告） |

- 自动投研模式的产出写入提案表，用户在各管理页面进行确认或拒绝
- 对话模式中 Agent 可读取当前用户的事实表数据（趋势、行业、股票、用户偏好）和新闻简报作为分析上下文
- 两种模式共享同一套 Agent 团队和配置

### 3.2 Agent 团队

系统内置以下 Agent，各自承担独立的分析职责：

| Agent | 职责 |
|-------|------|
| 每日新闻分析师 | 每日定时总结前一日新闻，产出结构化新闻简报，供后续投研和对话引用 |
| 舆情监控分析师 | 定时监控突发新闻和舆情事件，判断相关性，高相关时自动触发趋势/行业/股票分析流程 |
| 宏观分析师 | 分析新闻流、股市大盘、行业指数、大宗商品走势，产出关键趋势提案 |
| 行业机会分析师 | 基于宏观趋势结合近期新闻，深度挖掘有投资机会的行业，产出关注行业提案 |
| 行业分析师 | 对行业进行深度调研，出具行业调研报告 |
| 行业股票分析师 | 根据行业调研报告，在行业内选择具有投资价值的股票，提出关注股票提案 |
| 技术分析师 | 对具体股票走势进行技术分析，输出技术分析报告 |
| 社交媒体情绪分析师 | 获取投资者情绪数据，评估新闻事件对股价的影响程度，输出情绪分析报告 |
| 新闻分析师 | 对相关新闻事件进行深度分析，评估对股价的影响，输出新闻分析报告 |
| 基本面分析师 | 获取财务数据，分析财报信息，输出估值分析报告和基本面投资建议 |
| 看涨分析师 | 基于分析报告及历史记忆，构建看涨论据，同时反驳看跌观点 |
| 看跌分析师 | 基于分析报告及历史记忆，构建看跌论据，同时反驳看涨观点 |
| 研究经理 | 根据分析报告、辩论记录和历史记忆，给出买入、卖出或持有的结论 |
| 交易员 | 根据投资计划和分析报告，输出含有目标价位、置信度、风险评估的交易建议 |
| 激进分析师 | 从激进投资视角参与风险辩论，主张高风险高回报策略 |
| 保守分析师 | 从保守投资视角参与风险辩论，强调风险规避和安全边际 |
| 中立分析师 | 从中立视角参与风险辩论，平衡收益与风险 |
| 风控经理 | 根据辩论结果、全部报告、历史记忆和交易计划，给出最终的股票买卖提案 |

### 3.3 自动投研工作流

自动投研模式按以下阶段顺序执行，Agent 之间的依赖关系和数据流向硬编码为固定工作流，**用户不可修改流程**。运行过程中应能实时查看各 Agent 的执行状态和中间产出。

**阶段一：趋势发现**
- **宏观分析师**从新闻流和关键行情走势中提炼关键趋势，产出趋势提案

**阶段二：行业筛选**
- **行业机会分析师**基于关键趋势结合近期新闻进行深度挖掘，产出关注行业提案

**阶段三：行业调研与选股**
- **行业分析师**对关注行业进行深度调研，产出行业调研报告
- **行业股票分析师**根据调研报告在行业内选择具有投资价值的股票，提出关注股票提案

**阶段四：多维分析**（以下四个 Agent 并行执行）
- **技术分析师**：出具技术分析报告
- **社交媒体情绪分析师**：出具情绪分析报告
- **新闻分析师**：出具新闻分析报告
- **基本面分析师**：出具基本面分析报告

**阶段五：投资辩论**
- **看涨分析师**与**看跌分析师**基于四份研究报告及历史记忆进行投资辩论

**阶段六：投资决策**
- **研究经理**根据分析报告、辩论记录和历史记忆，得出买入、卖出或持有的结论

**阶段七：交易计划**
- **交易员**根据投资计划和分析报告，输出含有目标价位、置信度、风险评估的交易建议

**阶段八：风险辩论**（以下三个 Agent 并行执行）
- **激进分析师**、**保守分析师**、**中立分析师**各自发表核心观点，并针对其他方的论点进行反驳
- 三方不要求达成共识，最终产出一份简要的**风险观点报告**，汇总三方立场和核心论据，交由风控经理评估

**阶段九：风控决策**
- **风控经理**根据辩论结果、全部报告、历史记忆和交易计划，给出最终的股票买卖提案

### 3.4 投研产出

#### 对话模式产出

- 以自然语言回复用户提问，附带相关数据和分析依据
- 可引用当前用户的事实表数据（趋势、行业、股票状态）作为上下文
- 支持多轮追问，逐步深入分析

#### 自动投研模式产出

自动投研模式按 3.3 定义的工作流逐步执行，产出以下内容：

- **趋势提案**（阶段一）：宏观分析师产出的关键趋势增删改提案
- **行业提案**（阶段二）：行业机会分析师产出的关注行业增删改提案
- **行业调研报告**（阶段三）：行业分析师的深度调研报告
- **股票提案**（阶段三）：行业股票分析师产出的关注股票增删改提案
- **四维分析报告**（阶段四）：技术分析、情绪分析、新闻分析、基本面分析四份报告
- **投资辩论记录**（阶段五）：看涨与看跌分析师的辩论过程
- **投资结论**（阶段六）：研究经理的买入/卖出/持有结论
- **交易建议**（阶段七）：交易员的目标价位、置信度、风险评估
- **风险观点报告**（阶段八）：激进、保守、中立三方的立场和核心论据汇总
- **最终股票买卖提案**（阶段九）：风控经理综合所有信息给出的最终提案

所有提案写入提案表，用户在各管理页面进行确认或拒绝后更新事实表。

### 3.5 中间报告保留

自动投研过程中各阶段产出的中间报告应完整保留，包括：

- 行业调研报告（行业分析师产出）
- 技术分析报告（技术分析师产出）
- 情绪分析报告（社交媒体情绪分析师产出）
- 新闻分析报告（新闻分析师产出）
- 基本面分析报告（基本面分析师产出）
- 投资辩论记录（看涨/看跌分析师产出）
- 风险观点报告（激进/保守/中立分析师产出，含三方立场和核心论据）

中间报告关联到对应的投研运行和提案，用户在审核提案时可查看相关的全部中间报告作为决策参考。

## 4. Agent 配置

每个用户可独立管理投研团队中各 Agent 的**行为配置**，但**不能修改工作流程**（DAG 拓扑、任务依赖、数据流向固定不变）。

所有默认值硬编码在系统中。用户在管理页面的修改单独存储，运行时优先使用用户自定义值；如果用户未自定义某项配置，则回退使用系统默认值。

### 4.1 系统提示词管理

- 每个 Agent 的默认系统提示词硬编码在系统中
- 用户可在管理页面查看默认提示词并编辑覆盖
- 用户未自定义时，投研运行使用系统默认提示词

### 4.2 技能与工具配置

- 系统内置所有技能（Skill）和工具（Tool），用户不可创建、编辑或删除
- 每个 Agent 默认关联一组技能和工具（硬编码在系统中）
- 用户可在 Swarm 配置页面为每个 Agent 选择要启用的技能和工具（从系统内置列表中选择）
- 用户未自定义时，投研运行使用系统默认配置

### 4.3 管理边界

**用户可修改**（按用户隔离）：
- 各 Agent 的系统提示词
- 各 Agent 启用的技能（从内置列表中选择）
- 各 Agent 启用的工具（从内置列表中选择）

**用户不可修改**（全局硬编码）：
- Agent 的数量和角色定义
- Agent 之间的依赖关系（DAG 拓扑）
- 任务的执行顺序和数据流向
- 技能和工具本身的内容和定义

## 5. 数据管理

### 5.1 提案机制

趋势、行业、自选股的状态变更均通过**提案（Proposal）**触发，用户或 AI 不能直接修改事实表状态。

**提案操作类型：**

- **新增**：创建提案同时在事实表新增一条 `proposed` 记录，**立即生效**
- **更新**：事实表记录变为 `proposed`，保留原值用于拒绝时回滚
- **删除**：事实表记录状态不变，用户确认后才更新为 `removed`

**提案应记录的信息：**
- 目标类型（趋势/行业/股票）、目标记录、操作类型
- 提案状态：`pending`（待审核）→ `adopted`（已采纳）或 `rejected`（已拒绝）
- 提案标题、变更摘要
- 目标值（变更后的字段值）、原始值（仅更新类型，用于回滚）
- 来源信息（关联的投研运行和产出 Agent）
- 创建时间和审核时间

**通用规则：**
- 每个用户可自行设置各类型（趋势/行业/股票）待审批**新增**提案的数量上限（系统默认 10 条）
- 达到上限后，新的新增提案会替换当前置信度最低的待审批新增提案（置信度最低的被自动撤回）
- **更新**和**删除**类型的提案不受数量上限限制
- 上限机制的目的是帮助 Agent 保持专注，避免注意力过于分散，确保每次投研产出的提案数量可控、质量集中
- 不设提案审批超时，通过置信度淘汰机制管理待审批提案
- 系统应维护**提案审计日志**，记录所有变更操作的历史（谁、何时、把什么、变成了什么状态）

### 5.2 趋势管理

系统通过**事实表 + 提案**机制管理宏观趋势：

- 趋势按时间维度分为三级：**长期**（>1年）、**中期**（3个月~1年）、**短期**（<3个月）
- 每条趋势记录包含：级别、置信度（0~10 分）、支撑依据（相关新闻、数据指标、事件等）
- 趋势状态流转：`proposed` → `adopted` → `removed`，或 `proposed` → `rejected`

| 状态 | 说明 |
|------|------|
| `proposed` | AI 新增或更新的提案，**立即生效**参与后续分析 |
| `adopted` | 用户确认采纳 |
| `rejected` | 用户拒绝 |
| `removed` | 用户确认删除已采纳的记录 |

- 用户可在趋势页面直接审核 AI 提案（确认/拒绝）
- 用户可手动添加趋势（直接为 `adopted`），添加后自动触发完整的自动投研流程（从阶段一起运行）
- 活跃趋势的判定：`proposed` 和 `adopted` 均为活跃状态

### 5.3 行业管理

系统通过**事实表 + 提案**机制管理关注行业：

- 每条行业记录包含：置信度（0~10 分）、入选理由、行业调研报告、推荐股票
- 状态流转与趋势相同：`proposed` → `adopted` → `removed`，或 `proposed` → `rejected`
- 用户可在行业页面直接审核 AI 提案（确认/拒绝）
- 用户可手动添加行业（直接为 `adopted`），添加后自动触发完整的自动投研流程（从阶段二起运行）
- 活跃行业的判定：`proposed` 和 `adopted` 均为活跃状态

### 5.4 自选股管理

系统通过**事实表 + 提案**机制管理用户自选股票：

- 每条股票记录包含：股票名称、代码、置信度（0~10 分）、所属行业、持仓、操作建议（买入/卖出/持有）、目标价位、止损位、理由
- 状态流转与前两者相同：`proposed` → `adopted` → `removed`，或 `proposed` → `rejected`
- 用户可在自选股页面直接审核 AI 提案（确认/拒绝）
- 用户可手动添加股票（直接为 `adopted`），添加后自动触发完整的自动投研流程（从阶段四起运行）
- 活跃股票的判定：`proposed` 和 `adopted` 均为活跃状态

## 6. 后台任务

### 6.1 每日新闻存档

- 系统每日定时自动运行**每日新闻分析师**，总结前一日新闻并归档存档
- 存档的新闻简报供后续投研运行和对话模式引用，作为分析上下文的一部分
- 运行时间可由用户配置（默认每日早上 8:00），用户无感知，后台自动完成
- 历史新闻简报应持久化存储，可追溯查看

### 6.2 突发舆情监控

- 系统定时自动运行**舆情监控分析师**，实时监控突发新闻和舆情事件
- 舆情监控分析师判断事件的**相关性**（是否影响当前用户的关注趋势、行业或股票）
- 当判断为高相关性事件时，立即运行相关的自动投研流程以实现快速响应：
  - 影响趋势 → 从阶段一起运行完整流程
  - 影响行业 → 从阶段二起运行
  - 影响持仓股票 → 从阶段四起运行（跳过行业调研和选股）
- 产出的提案正常推送（Web 端和飞书），用户按正常流程审批
- 监控频率可由用户配置（默认每小时一次），用户无感知，后台自动完成
- 舆情事件和触发的分析结果应记录存档，可追溯查看

## 7. 用户偏好

每个用户可设置自己的投资偏好，Agent 在进行研究分析时应参考用户的偏好配置：

| 偏好项 | 说明 | 示例 |
|--------|------|------|
| 投资风格 | 用户的投资策略偏好 | 价值投资、成长投资、均衡配置 |
| 风险偏好 | 用户的风险承受能力 | 保守型、稳健型、积极型 |
| 关注市场 | 用户主要关注的市场 | A 股、港股、美股、加密货币 |
| 关注行业 | 用户偏好的行业方向 | 科技、消费、医药、金融 |
| 持仓周期 | 用户的典型持仓时间 | 短线（日内~1 周）、中线（1 周~3 月）、长线（3 月以上） |
| 资金规模 | 用户的投资资金量级 | 10 万以下、10~50 万、50~100 万、100 万以上 |
| 股票投资总额 | 用于股票投资的实际资金金额，用于仓位评估 | 具体金额（如 50000） |
| 避险标的 | 用户希望规避的行业或标的 | ST 股、次新股、高杠杆标的 |
| 自定义备注 | 用户的其他投资要求或偏好描述 | 自由文本 |

- 偏好数据按用户隔离存储
- 投研运行时，Agent 可读取当前用户的偏好，作为分析上下文的一部分
- 用户未设置的偏好项不参与分析，不影响默认行为

## 8. 数据源

系统应支持以下数据源获取金融行情和搜索信息：

| 数据源 | 用途 |
|--------|------|
| baostock | A 股行情数据（K线、指标等） |
| tushare | A 股行情、财务数据 |
| akshare | A 股行情（免费备用）、宏观数据 |
| searxng | 搜索引擎聚合，用于新闻流和信息检索 |

系统应支持数据源的灵活切换和降级回退（如 tushare 不可用时自动使用 akshare）。

## 9. 外部集成

### 9.1 飞书集成

系统应支持与飞书（Lark）对接，提供移动端的提案审批和对话能力：

#### 9.1.1 提案推送与审批

- 新提案产生时，系统应通过飞书机器人向对应用户推送消息卡片，展示提案摘要（标题、类型、置信度、变更摘要）
- 用户可在飞书消息卡片上直接操作**采纳**或**拒绝**，操作结果同步回系统，更新事实表状态
- 支持**趋势**、**行业**、**自选股**三种类型提案的推送与审批

#### 9.1.2 飞书对话

- 用户可通过飞书与 Agent 进行**对话模式**的交互（即 3.1 中定义的对话模式）
- 对话上下文与 Web 端共享，用户可在飞书追问、调整分析方向
- Agent 回复以飞书消息形式返回，支持文本和简单卡片格式

#### 9.1.3 用户配置

- 每个用户可自行绑定飞书账号，配置推送渠道（个人/群聊）
- 飞书集成所需的 App ID、App Secret 等由用户自行配置

## 10. 非功能需求

### 10.1 性能

- 单次完整投研分析（从启动到输出报告）应在合理时间内完成
- 多个无依赖关系的 Agent 应能并行执行

### 10.2 可靠性

- Agent 执行失败时应有重试机制
- 用户应能查看执行失败的 Agent 的错误信息

### 10.3 可扩展性

- 应支持新增 Agent 角色
- 应支持新增数据源
- 应支持新增 Skill

## 11. 约束

- 系统应支持本地部署
- 系统应基于当前 Vibe-Trading 项目构建
- 用户认证和所有配置信息应本地存储，不依赖外部认证服务
- API Key 等敏感信息应加密存储，前端展示时脱敏处理
- 系统不提供默认 API Key，所有第三方服务密钥由用户自行配置
